The **2026 Workflows engine** rebuilt Attio's automation around AI agent steps, native JavaScript, and a credit model where most blocks are free. Attio shipped it in June 2026 and closed the previous engine to new creation: you can no longer create new legacy workflows; existing ones keep running, and Attio says sunset notice will come "well in advance." This is not a facelift. It is a different product with a different pricing model, a larger block library, and a builder that treats AI as a first-class step rather than a bolt-on.

For a GTM (go-to-market) engineer, the practical consequence is that logic you used to push out to Zapier, Make, or Clay can now live inside Attio, beside the records it reads and writes. An [AI agent](/reference/agents/custom-agent) can research a lead and draft a follow-up in the same run that an [Execute code](/reference/calculations/execute-code) block reshapes a webhook payload, and the read-only, logic-only parts of that run cost nothing.

This page explains what changed since the legacy engine and how each change affects the way you build. After reading it, you can decide whether to migrate an existing workflow, recognize which new blocks unlock work you could not do before, and reason about the new credit model before you ship anything to production.

## Running agents as workflow steps

The headline change is that AI agents now run inside a workflow as ordinary steps, so an agent can read context, decide, and hand its output to the next block in the same run. The [Custom agent](/reference/agents/custom-agent) block configures a model — GPT, Sonnet, Opus, or Gemini Pro — with an instruction prompt and a set of tools. Its access to your workspace is **read-only**, gated by the **"Run as"** permission model, so an agent can read records, notes, emails, and call recordings but cannot write or delete on its own; you place an explicit write block after it when you want a change persisted.

A minimal agent step needs three things: a model, a prompt, and the record context it should read.

```json
{
  "block": "custom_agent",
  "model": "Sonnet",
  "run_as": "ops-bot@acme.com",
  "instructions": "Summarize the latest three notes on this company and flag any churn risk in one sentence.",
  "context": "{{ trigger.record }}"
}
```

The agent returns a text result you reference downstream as `{{ steps.custom_agent.output }}` and feed into an [Update record](/reference/records/update-record) write or a Slack message. The [Web agent](/reference/agents/web-agent) block is the former "Research record" block rebuilt: you ask a question about a record and the agent researches across the web to answer it. You can also describe a workflow in plain language in the canvas and let Ask Attio assemble it block by block, live.

## Writing custom logic with native code

The [Execute code](/reference/calculations/execute-code) block runs arbitrary JavaScript as a step, which removes the external webhook round-trip that data shaping used to require. You export a `main(inputs)` function, and whatever it returns becomes the block's output. Transformations, custom scoring, and payload reshaping now run inline, beside the records they touch, with a 100-second timeout per run.

```javascript
export function main(inputs) {
  const domain = inputs.email.split("@")[1];
  const tier = inputs.employees > 500 ? "enterprise" : "mid-market";
  return { domain, tier };
}
```

Given `jane@acme.com` and an employee count of 1,200, this block returns `{ "domain": "acme.com", "tier": "enterprise" }`, which a later block reads as `{{ steps.execute_code.tier }}`. Because the cost of [Execute code](/reference/calculations/execute-code) scales with the work it does, its credit cost is listed as **Variable (complexity-based)** rather than a flat per-run figure. For heavier integration logic, the [Send HTTP request](/reference/utilities/send-http-request) block pairs naturally with code to call an external API and parse the response in the same workflow.

## Paying for outcomes, not steps

The credit model was rebuilt so that you pay only when a block does something that costs Attio money to run. Triggers, conditions, lookups, and delays are free; credits are consumed only when a block writes data, sends or receives something externally, or invokes AI. The shift matters most for read-heavy, logic-heavy workflows, which used to charge a credit per step and now run their logic for nothing.

Consider a routing workflow that fires on a [Webhook received](/reference/triggers/webhook-received) trigger, reads three records, evaluates two conditions, and ends in a single write. Under the legacy engine every one of those steps consumed a credit; under the 2026 engine only the final write does, so the same workflow costs one credit per run instead of six or seven. AI blocks moved the other way: instead of a fixed 1–10 credits, they now bill by the tokens the model consumes, which makes an AI-heavy workflow variable-cost and worth metering before you scale it.

One legacy safety valve is gone: **per-workflow credit caps were removed** in the new engine, so you can no longer set a hard credit ceiling on an individual workflow. Budget at the workspace level and watch run history instead. The full per-block table, plan allowances, and guidance on reasoning about variable cost live on [Credits & pricing explained](/reference/credits-and-pricing).

## Building with the new canvas

The builder gained the quality-of-life features that made the legacy engine tedious for anything beyond a single trigger. A workflow can now start from several events at once — multi-trigger workflows — and you resolve whichever event fired using the fallback patterns described in [Variables & data passing](/explanation/variables). You can copy and paste blocks between workflows and even between workspaces, which turns a tested pattern in one client's workspace into a starting point in another's.

In-flight runs are now recoverable: you can retry, pause, and resume them, and re-run a failed step without restarting the whole workflow, as covered in [Runs, debugging & publishing](/explanation/runs-and-debugging). Blocks drag in from the sidebar, HTTP payloads have a visual JSON editor, and you can drop a block straight into a loop. The library itself is extensible — you can ship your own blocks with the [App SDK custom blocks](/guides/platform-patterns/app-sdk-custom-blocks) workflow, and Attio added ecosystem blocks for Webflow, Notion, Linear, and more.

## Deciding whether to migrate

If you have legacy workflows running, they keep running, and Attio says sunset notice will come "well in advance" — but new capability and the cheaper pricing exist only in the new engine. Several blocks were renamed and a few behave differently, so work through [Migrate legacy workflows](/guides/migrating-from-legacy) and check the [Block changes reference](/guides/migrating-from-legacy/block-changes) before you rebuild anything.

If you are starting fresh, ignore legacy entirely and learn the new engine's [Block reference overview](/reference) and the handful of core ideas in [Choosing the right trigger](/explanation/triggers) that everything else hangs off. The judgment call is no longer *can* Attio host a piece of logic but *should* it: the [Guides](/guides) section shows where workflows genuinely win against a standalone automation tool.
