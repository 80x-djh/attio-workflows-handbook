The **Create or update record** block (Records) matches on a unique attribute, updates the record if found, or creates one if not. It is the idempotent upsert step for records, so a webhook or trigger that fires twice for the same person or company produces one record instead of two. The match attribute must be genuinely unique, such as `email_address` for a person or `domains` for a company, or the block can silently overwrite the wrong record.

## Inputs

| Input | Type | Required | Default | Notes |
| --- | --- | --- | --- | --- |
| Object | Object | Yes | — | The object to create or update a record in, such as People or Companies. |
| Matching attribute | Attribute | Yes | — | The unique attribute used to find an existing record. Choose an attribute that identifies one record only, such as email or domain. |
| Attributes to set | Attribute values | No | — | The values to write. On a match these update the existing record; with no match they populate the new record. |
| Replace existing values | Boolean | No | Off | Controls multi-select behavior. When on, the block replaces the existing values; when off, it appends to them. |

## Outputs

| Output | Type | Example | Notes |
| --- | --- | --- | --- |
| Record | Record | The Priya Sharma person record | The created or updated record, with its attribute values after the write. |
| Record ID | Text | `a1b2c3d4-...` | The stable identifier of the matched or newly created record. |

Reference the resulting record's ID in a later block as `{{ steps.create_or_update_record.record_id }}`.

## Behavior

The block reads the matching attribute first, then decides whether to update or create. If exactly one record has the same value on the matching attribute, the block updates that record. If no record matches, the block creates a new one and populates it from the **Attributes to set** input. The output is the same shape in both cases, so a downstream block does not need to know which branch ran.

Single-value attributes always overwrite. Writing a new email or stage replaces the previous value regardless of the **Replace existing values** setting, which only governs multi-value attributes.

Multi-value attributes follow the **Replace existing values** toggle. With the toggle off, the block appends the new values to whatever is already on the record. With the toggle on, the block discards the existing values and writes only the new set. This setting is the single most common cause of unexpected data loss with this block, because turning it on for a `tags` or `categories` attribute deletes every value the block did not explicitly include.

The block writes on every run. It does not deduplicate beyond the single matching attribute, so a record that matches on email still updates even if other attributes differ.

## Gotchas

- **The wrong record gets overwritten.** A non-unique matching attribute, such as a first name or a company stage, can match several records, and the block updates one of them rather than the one you meant. Match on an attribute that is unique per record, such as `email_address` or `domains`.
- **Multi-select values disappear.** Turning on **Replace existing values** for a multi-value attribute deletes every existing value that the run does not re-supply. Leave the toggle off to append, and turn it on only when you intend to replace the full set.
- **The block updates with stale data.** Reading from the trigger payload after an earlier block has already changed the record writes outdated values. Reference the output of the block that made the change, not the original trigger data.
- **The run fails on a write without access.** A workflow has no write access to objects by default, so the block errors until you grant the workflow write access to the target object. Grant access before you publish.

## Examples

### Upserting an inbound person by email

A webhook lead-intake workflow receives a form submission and needs to create the person once, then update them on every later submission.

| Field | Value |
| --- | --- |
| Object | People |
| Matching attribute | `email_address` |
| Attributes to set | `name` = `Priya Sharma`, `job_title` = `VP of Revenue`, `company` = `acme.com` |
| Replace existing values | Off |

The first submission from `priya@acme.com` creates the person record. A second submission with an updated job title finds the same record by email and updates it in place, leaving the workspace with one Priya Sharma record.

### Appending a source tag without losing existing tags

A routing workflow tags inbound people with the channel they arrived on while preserving the tags they already carry.

```json
{
  "object": "People",
  "matching_attribute": "email_address",
  "attributes_to_set": {
    "lead_source": ["Webinar"]
  },
  "replace_existing_values": false
}
```

A person who already has `["Newsletter"]` on `lead_source` ends up with `["Newsletter", "Webinar"]`, because the toggle is off and the block appends rather than replaces.

### Resetting a multi-select to a single value

A cleanup workflow needs the `lifecycle_stage` tags on a company to reflect exactly one current state, discarding prior values.

| Field | Value |
| --- | --- |
| Object | Companies |
| Matching attribute | `domains` |
| Attributes to set | `lifecycle_stage` = `Customer` |
| Replace existing values | On |

The company matched on `acme.com` keeps only `Customer` on `lifecycle_stage`, and any earlier values such as `Prospect` or `Trial` are removed.

## Credit cost worked example

The block costs 1 credit per run, charged whether it creates a record or updates one. A webhook lead-intake workflow that upserts 800 inbound people a month spends 800 credits on this block. Triggers, conditions, and lookups in the same workflow are free, so the upsert is the only paid step in a typical intake flow.
