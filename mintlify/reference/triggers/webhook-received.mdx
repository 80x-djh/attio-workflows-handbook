The **Webhook received** trigger (external) starts a workflow when an external system sends JSON to the workflow's webhook URL. It is the entry point for any event that originates outside Attio, such as a product event, a form submission, or a partner lead. The payload arrives as untrusted data, so the first block after the trigger is almost always [Parse JSON](/reference/utilities/parse-json).

| | |
| --- | --- |
| **Category** | Trigger (external) |
| **Credit cost** | Free — triggers consume no credits |
| **Engine** | New Workflows engine (June 2026+) |
| **Plan availability** | All plans |
| **Limits** | Requests must use `Content-Type: application/json`. Outputs are limited to the payload body and a timestamp; request headers are not exposed. |

## Inputs

The trigger generates a webhook URL when you add it to a workflow. You configure the external system rather than the trigger itself, so there are no fields to map.

| Input | Type | Required | Default | Notes |
| --- | --- | --- | --- | --- |
| Webhook URL | URL (read-only) | — | Generated by Attio | Copy this URL into the sending system. The system must send an HTTP request with the header `Content-Type: application/json`. |

## Outputs

The trigger exposes the raw request body and the time the request arrived. Request headers are not available as outputs, so you cannot read authentication tokens or custom headers downstream.

| Output | Type | Example | Notes |
| --- | --- | --- | --- |
| Payload body | Text (raw JSON) | `{"email":"jane@acme.com","company":"acme.com"}` | The unparsed request body. Pass it to [Parse JSON](/reference/utilities/parse-json) before reading fields. |
| Timestamp | Datetime | `2026-06-11T14:32:05Z` | The moment Attio received the request. |

Reference the raw body in a later block as `{{ steps.webhook_received.body }}` and the arrival time as `{{ steps.webhook_received.timestamp }}`.

## Behavior

The trigger fires once per incoming HTTP request that reaches the webhook URL. Attio passes the request body through as raw JSON text; it does not parse, validate, or coerce the payload, so every field arrives exactly as the sender wrote it.

A request without the `Content-Type: application/json` header does not produce a usable run. Treat the payload as untrusted: a sending system can omit an optional field, send a number where you expected a string, or change its schema without warning. Because the body is raw text, downstream blocks see no structure until you parse it.

## Gotchas

- **Fields read as empty until you parse the body.** The trigger output is raw JSON text, not structured data, so a mapping such as `{{ steps.webhook_received.email }}` resolves to nothing. Follow the trigger with [Parse JSON](/reference/utilities/parse-json), then map fields from the parsed result.
- **The request silently does nothing without the JSON content type.** A sender that posts form-encoded data or omits the `Content-Type: application/json` header will not produce a usable payload. Set the header on the sending system before testing.
- **A malformed or unexpected payload corrupts records downstream.** External systems change their schema without notice, and a missing or wrong-typed field can write blank or incorrect values. Validate the parsed payload with a [Filter](/reference/conditions/filter) before any block that creates or updates a record.
- **Authentication headers are invisible to the workflow.** The trigger exposes only the body and a timestamp, so you cannot verify a signature or bearer token from a request header. Carry any shared secret inside the JSON body and check it with a [Filter](/reference/conditions/filter).

## Examples

### Capturing a product signup

A product backend posts a signup event to the webhook URL whenever a new account is created.

```json
{
  "email": "jane@acme.com",
  "full_name": "Priya Sharma",
  "company_domain": "acme.com",
  "plan": "pro"
}
```

Attio starts a run with this body available as `{{ steps.webhook_received.body }}`, ready for [Parse JSON](/reference/utilities/parse-json) to expose `email`, `full_name`, and `company_domain`.

### Routing a partner lead with a shared secret

A referral partner posts a lead and includes a shared secret in the body so the workflow can confirm the source before writing anything.

```json
{
  "secret": "hta-partner-2026",
  "email": "marcus.lee@northwind.io",
  "source": "northwind-referral"
}
```

After [Parse JSON](/reference/utilities/parse-json), a [Filter](/reference/conditions/filter) requiring `secret` to equal `hta-partner-2026` blocks any request that lacks the value, and only verified leads reach the [Create or update record](/reference/records/create-or-update-record) block.
