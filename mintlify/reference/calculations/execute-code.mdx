The **Execute code** block (Calculations) runs custom JavaScript inside a workflow and exposes the value it returns to later blocks. Reach for it when a [Formula](/reference/calculations/formula) or [Aggregate values](/reference/calculations/aggregate-values) block cannot express the transformation you need, such as reshaping a payload, deriving a score from several attributes, or normalizing inconsistent source data. The block runs your code with a 100-second timeout, so it suits short transformations rather than long-running jobs.

| | |
| --- | --- |
| **Category** | Calculations |
| **Credit cost** | Variable (complexity-based) |
| **Engine** | New Workflows engine (June 2026+) |
| **Plan availability** | Not documented. |
| **Limits** | 100-second execution timeout. |

## Inputs

| Input | Type | Required | Default | Notes |
| --- | --- | --- | --- | --- |
| Code | JavaScript | Yes | — | Must export a function named `main` that receives a single `inputs` object. The function can be synchronous or asynchronous. |
| Variables | Any | No | — | Workflow variables you map into the `inputs` object. Each becomes a key on `inputs`, addressable as `inputs.<key>` inside `main`. |

The block passes every variable you map into the block as one `inputs` object. Whatever `main` returns becomes the block's output. A minimal function reads its inputs and returns a plain object:

```js
export async function main(inputs) {
  return { score: inputs.signals * 10 };
}
```

## Outputs

| Output | Type | Example | Notes |
| --- | --- | --- | --- |
| Result | Any | `{ "score": 70 }` | The exact value returned by `main`. Return an object to expose several named fields downstream. |

Reference the returned value in a later block as `{{ steps.execute_code.result }}`. If `main` returns an object, address each field by key, such as `{{ steps.execute_code.result.score }}`.

## Behavior

The block invokes the exported `main` function once per run and waits for it to resolve before continuing. An `async` function is awaited, so you can `await` inside `main`, but the work must finish within the 100-second timeout. Code that exceeds the timeout fails the block rather than returning a partial result.

The return value passes through verbatim, including `null`, an empty object, or an empty array. Downstream blocks receive exactly what `main` returned, so a missing field is `undefined` rather than an error. Because the credit cost is variable and complexity-based, heavier code costs more than a light transformation; keep functions small and single-purpose.

## Gotchas

- **The function must be named `main`.** The block calls `main(inputs)` by name, so an unnamed function, a differently named export, or a default export fails to run. Export `main` exactly, and pass your inputs through its `inputs` parameter rather than reading any outer scope.
- **A thrown error fails the whole block.** Any uncaught exception inside `main` stops the run at this block, so a single malformed field can halt the workflow. Wrap risky parsing in a `try`/`catch` and return a sentinel value, such as `{ error: true }`, that a later [If](/reference/conditions/if) block can branch on.
- **Non-deterministic code makes runs hard to trust.** When the result drives a write, calls to `Date.now()`, `Math.random()`, or an external service can produce a different value on every run and on every retry. Keep code that controls writes deterministic, and move randomness to the [Random number](/reference/calculations/random-number) block so the value is captured in run history.
- **Long jobs hit the 100-second timeout.** The block stops execution at 100 seconds and fails, so polling loops and large batch loops are the wrong fit. Split the work across a [Loop](/reference/utilities/loop) block, or call an external service with [Send HTTP request](/reference/utilities/send-http-request).

## Examples

### Normalizing a lead source field

This function maps inconsistent source labels onto a single canonical value before the workflow writes the record.

```js
export async function main(inputs) {
  const raw = (inputs.source || "").toLowerCase().trim();
  const map = {
    "google ads": "Paid search",
    adwords: "Paid search",
    linkedin: "Social",
    "li ads": "Social",
  };
  return { source: map[raw] || "Other" };
}
```

With `inputs.source` set to `LI Ads`, the block returns `{ "source": "Social" }`, which an [Update record](/reference/records/update-record) block writes to the lead's source attribute as `{{ steps.execute_code.result.source }}`.

### Scoring a record from several attributes

This function combines three mapped variables into one explainable score for a deal owned by Priya Sharma.

| Field | Value |
| --- | --- |
| Code | the `main` function in this example |
| `inputs.employees` | `220` |
| `inputs.uses_competitor` | `true` |
| `inputs.opened_email` | `true` |

```js
export async function main(inputs) {
  let score = 0;
  if (inputs.employees >= 200) score += 40;
  if (inputs.uses_competitor) score += 20;
  if (inputs.opened_email) score += 10;
  return { score };
}
```

The block returns `{ "score": 70 }`, which a downstream [If](/reference/conditions/if) block reads as `{{ steps.execute_code.result.score }}` to route deals scoring 50 or higher to a sales-ready stage.

## Credit cost worked example

Execute code is priced as **Variable (complexity-based)**, so a single fixed-per-run figure does not apply; a heavier function costs more than a light one. Estimate spend by sampling a few real runs in run history, then multiplying the observed cost by your monthly volume. For a lead-normalization workflow that fires on 2,000 inbound leads a month, measure the cost of one representative run, multiply by 2,000, and compare the total to your monthly allowance on [Credits & pricing explained](/reference/credits-and-pricing). Logic and lookup blocks elsewhere in the same workflow stay free, so only the Execute code step and any write blocks contribute to the total.
